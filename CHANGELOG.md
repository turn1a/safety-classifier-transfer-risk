# Changelog

All notable changes to this project are documented here. The format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and the project aims to follow [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added

- Project scaffold: a Kedro 1.4 project (`transfer_risk`) with eight modular pipelines — data, models, similarity, attacks, transfer, risk, reporting, plus a `smoke` wiring check — over a typed Data Catalog with kedro-viz layer metadata.
- Pure-core package `transfer_risk.lib`: linear/minibatch CKA, Diagonal Box Similarity, deterministic seeding, threshold calibration, plus the transfer-rate and ablation statistics — implemented and unit-tested against real invariants (90% coverage gate).
- The seven domain pipelines (data, models, similarity, attacks, transfer, risk, reporting) implemented end-to-end: HuggingFace data harmonisation, surrogate fine-tuning + the from-scratch BiLSTM, CKA/DBS similarity, the TextAttack sweep, transfer evaluation, and the risk regression + ablation.
- Raw training sources pulled through `HFDataset` catalog entries; trained surrogates persisted through a custom `SurrogateModelDataset` (the `surrogate.{name}` factory).
- MLflow tracking of flattened params, run metrics (transfer rate, similarity-vs-transfer Spearman, ablation effect/p-value, calibrated thresholds), and artifacts (figures, tables); rich console + rotating-file logging (`conf/logging.yml`); `.env`-based runtime config; an `interrogate` docstring gate (public + private); and `just setup-data` for the offline NLP assets.
- House-style tooling: uv (`uv_build`), ruff, mypy strict, pytest + coverage, two-stage pre-commit with nbstripout, and a justfile.
- Experiment tracking via kedro-mlflow on a local SQLite backend (`sqlite:///mlflow.db`; MLflow 3.x deprecates the `./mlruns` file store).
- A Quarto documentation site (the blog series), published to GitHub Pages by CI.
- GitHub Actions: lint/type/test on Ubuntu (with a gated macOS-14 job) and a docs publish workflow.
- Results blog post (second in the series) reporting the headline measurement against `protectai/deberta-v3-base-prompt-injection-v2`: CKA similarity predicts attack transfer (Spearman ρ = 0.72 per attack, 0.79 per surrogate), the same-backbone anchor tops both CKA and transfer, and the high-similarity surrogate half transfers ~3× the low-similarity half. Run scope (38 of 50 cells, all four perturbation families) and the inconclusive packaged ablation are reported honestly.
- Per-node timing via a `NodeTimingHook` (Kedro's `before/after_node_run` + `before/after_dataset_loaded/saved` hooks): one line per phase — `load <dataset> <s>`, `node <name> <s> (compute)`, `save <dataset> <s>` — on the `transfer_risk.timing` logger (set it to WARNING in `conf/logging.yml` to silence). It separates S3 I/O from greedy-search compute, and because each `ParallelRunner` worker registers its own hook instance (Kedro re-applies `conf/logging.yml` per worker), the timings are emitted from the parallel workers too, not just the main process.
- A cloud path for the bottleneck stage. `infra/` is Terraform for a single high-core ARM Graviton spot box (S3 state-exchange bucket, least-privilege instance role, egress-only security group with SSM Session Manager access instead of SSH, IMDSv2), driven by `just cloud-stage / cloud-up / cloud-finish / cloud-logs / cloud-attach / cloud-down`. The box runs only the attack sweep (`--env cloud`: all cores via `ParallelRunner`, ONNX victims) reading and writing S3 through the catalog; the downstream and MLflow run locally, so MLflow stays a single local writer and results land on the dev machine. An `arch` variable flips arm64/x86_64 for the ONNX fallback. See [infra/README.md](infra/README.md) for the one-time AWS setup. (See Changed for the catalog-native S3 path that replaced the original `aws s3 sync` design.)

### Changed

- The pipeline reads and writes S3 through the Kedro catalog instead of `aws s3 sync` on local paths. A set of `${globals:...}` roots (`conf/base/globals.yml`, overridden to `s3://` in `conf/cloud/globals.yml` via scoped `tr.bucket`/`tr.region` resolvers registered in `settings.py`) points only the boundary artifacts (eval splits, surrogate checkpoints, ONNX graphs, adversarial partitions) at the bucket; everything else stays local. The models, similarity, and attacks pipelines are generated dynamically — one node per surrogate, and one per `(surrogate, recipe, example-shard)` — so `ParallelRunner` owns parallelism and `kedro run --only-missing-outputs` owns resume, replacing the in-node `ProcessPoolExecutor`, the manual per-cell partition writes, and the directory-glob resume. Pretrained surrogates, fine-tune backbones, and the frozen target are now catalog datasets (`HuggingFaceHubModelDataset`), and every surrogate is persisted through fsspec-aware model-directory datasets (`SurrogateModelDataset`, `OnnxModelDataset`, built on the `kedro_datasets` `TensorFlowModelDataset` pattern), so no node calls `from_pretrained(hub_id)` / `save_pretrained` / `aws s3 sync` and the spot box needs no Hub access or HuggingFace token. ONNX export is an in-environment `torch.onnx.export` pipeline node, parity-gated against the torch checkpoint (the `optimum`/`uvx` `just export-onnx` script is removed). Cloud recipes are `just cloud-stage` / `cloud-up` / `cloud-finish` (replacing `cloud-push` / `cloud-pull`); `just run-thin` sets `KEDRO_ENV=thin` so the dynamic pipelines build the reduced thin structure. The box's IAM role drops the SSM/KMS HuggingFace-token permissions, since the token is no longer needed there. For ParallelRunner: the box sets `KEDRO_MP_CONTEXT=spawn` (fresh workers, so each attack node sets its CPU/thread/encoder env before importing textattack/torch), and a `before_pipeline_run` hook registers a cloudpickle reducer on `ForkingPickler` for any materialised dataset it cannot pickle — kedro validates the whole catalog for multiprocessing (kedro #3804) and kedro-mlflow's dynamic `MlflowArtifactDataset` classes are not picklable by the default pickler, so this keeps the MLflow integration intact while letting the sweep run in parallel. Worker count is not a CLI option: the runner is built `ParallelRunner(is_async=...)` and defaults `max_workers` to `os.cpu_count()`, and the sweep DAG is two topological layers (every shard, then every reduce), so `len(nodes) - len(grouped_nodes) + 1` far exceeds the core count and the effective worker count is just the box's vCPUs — the instance size is the parallelism knob. Each worker pins one thread (`OMP_NUM_THREADS=1`, `torch.set_num_threads(1)`, ORT `intra_op_num_threads=1`), so vCPU-many single-threaded workers saturate the box without oversubscription. The box also runs with `--async` so a node's S3 input loads (splits + victim) and its partition save overlap with compute.
- The selection ablation now contrasts the high-CKA half (M1) against the low-CKA half (M2) with a one-sided permutation test on the difference in group-mean transfer rate, evaluated on both each surrogate's mean-across-recipes and max-across-recipes transfer. It replaces the earlier bootstrap that compared the union M1 ∪ M2 against equal-size random subsets and read only the single best surrogate's transfer; that statistic answered the wrong question (a random subset of nine often catches a strong surrogate too) and came out inconclusive even when M1 transferred far more than M2. Label assignments are enumerated exactly when the groups are small, so the p-value is exact — three vs three has a coarse floor of 1/20 = 0.05, which the risk node and the report state. `transfer_risk.lib.ablation` now exposes `permutation_test` and `selection_ablation` (replacing `random_subset`/`ablation_statistics`); `conf/base/parameters_risk.yml` swaps `n_bootstrap` for `n_permutations`; `ablation_results.json` and the ablation figure report the M1/M2 transfer means with the mean- and max-transfer effect sizes (pp) and p-values.
- The `attacks` pipeline now runs TextAttack in-process in the main environment (transformers 5, Python 3.13) instead of shelling out to a pinned Python 3.11 subenv. It uses a minimal fork, [turn1a/TextAttack](https://github.com/turn1a/TextAttack) (branch `transformers-5-compat`), that makes the three top-level `flair` imports lazy and adds a torch-native sentence-transformers encoder in place of the TensorFlow Universal Sentence Encoder (`semantic_encoder: use` restores the original, behind the `textattack[tensorflow]` extra). All five recipes run — DeepWordBug, PWWS, TextFooler, BAE, BERT-Attack — none dropped for a missing dependency.
- The attack sweep parallelises each `(surrogate, recipe)` pair across CPU worker processes (`num_workers`, single-threaded each). The per-example greedy search is memory-bandwidth-bound on Apple Silicon: CPU gives ~2-3× over single-process and saturates near the perf-core count, and MPS was measured ~30% slower (the search issues many small forwards, where GPU kernel-launch overhead beats the batch gain) — so the sweep runs on CPU. Model loading is forced offline (`HF_HUB_OFFLINE` / `TRANSFORMERS_OFFLINE`) to drop the per-load HuggingFace round-trips, and the goal-function query batch is enlarged to amortise the model-weight stream.
- Attack tractability knobs: prompts are truncated to `max_prompt_chars` before the search (its cost scales with word count and the prompts are long-tailed), and `eval_set_size` / `query_budget` are tuned so the sweep completes in hours rather than days. These bound the search cost without changing any per-example prediction; raising them recovers full precision at higher cost. The runner cleans up worker processes on SIGINT/SIGTERM and writes results incrementally, so a sweep is resumable.
- `HF_HUB_DOWNLOAD_TIMEOUT` is set in `.env` so a stalled HuggingFace model download errors and retries instead of hanging the run indefinitely.
- The sweep now parallelises at the `(surrogate, recipe, example-shard)` level rather than per-cell, so a slow cell (a robust victim on a heavy recipe) no longer pins a single core — wall time becomes total-work / cores. A `shard_size` knob controls the split; the per-shard seed is offset by the example's absolute index, so a fixed `shard_size` + seed reproduces exactly (PWWS is shard-size-invariant; the RNG recipes can shift across `shard_size`, so keep it fixed within a comparison set).
- Each `(surrogate, recipe)` cell is persisted the moment it completes as a partition under a `PartitionedDataset`, so an interrupted sweep (including a reclaimed spot instance) resumes by skipping cells already on disk. `evaluate_transfer` consumes the partitions as lazy per-cell loaders, and the `thin` env writes to an isolated partition directory so a smoke run never contaminates a real sweep through resume.
- Optional ONNX Runtime victim inference (`attacks.use_onnx`, off by default) for ~2-3× faster CPU queries, served by a thin `ONNXModelWrapper` that imports only `onnxruntime` (never `optimum`). Verified at 100% argmax agreement and ~5e-8 max softmax difference against the torch checkpoint — the gate that justifies turning it on.

### Fixed

- The four DeBERTa surrogates are served from their torch checkpoint during the attack sweep, not their ONNX graph. DeBERTa-v2/v3's disentangled-attention ONNX graph triggers a "MatMul dimension mismatch" in the cloud box's aarch64 onnxruntime at every sequence length (HF transformers #18237), which aborted the sweep on the first DeBERTa cell once the `.model` fix let recipes build; macOS onnxruntime tolerates the graph, so the local suite never caught it. Fixed-length padding (`ONNXModelWrapper` now tokenises with `padding="max_length"`, so every query matches the traced 512-shape) was necessary but not sufficient — the aarch64 build rejects the graph regardless — so the DeBERTa victims are served from torch (`HuggingFaceModelWrapper`) via a `victim: torch` flag in `parameters_models.yml`. The standard-attention transformers (BERT/RoBERTa/ELECTRA/Llama-Guard) keep the faster ONNX victim, which runs correctly on the box. The torch victim is argmax-identical to the model's ONNX graph (still exported for CKA/parity; just not the attack victim for these four), and the `(surrogate, recipe, shard)` fan-out and resume are unchanged.
- `ONNXModelWrapper` now exposes `self.model` (its ONNX Runtime session). TextAttack's `GoalFunction.__init__` reads `model_wrapper.model.__class__` for its model/goal-function compatibility check, so the ONNX victim wrapper — which set only `self.session` — raised `AttributeError: 'ONNXModelWrapper' object has no attribute 'model'` at recipe-build time, on the first attack node. Every transformer surrogate is served through the ONNX wrapper, so this aborted the whole cloud sweep before any partition was written (the box stopped within minutes, zero adversarial output). Queries go through `__call__`, never this attribute, so exposing the session is enough; the validator only warns on the unrecognised class. A hermetic integration test now builds and runs a recipe against a real tiny ONNX victim — the path the earlier stub-based and `__call__`-only tests missed.
- Surrogate fine-tuning now loads backbones in fp32 explicitly (`dtype=torch.float32`). `microsoft/deberta-v3-base` ships a float16 checkpoint, and transformers>=5 loads in the checkpoint dtype by default, so training it in raw fp16 (no loss scaling) diverged to NaN within an epoch and silently produced a degenerate `deberta-base-ft-seed` surrogate (validation accuracy at the class baseline, all weights NaN). Other backbones ship fp32 and were unaffected, which is why only the same-backbone-as-target anchor was hit. The three inference load sites (CKA representations, the attack/transfer wrapper, and the persisted-checkpoint dataset) also force fp32, both for numerical consistency across the pool and because fp16 ops are slow or unimplemented on the CPU the attack sweep runs on.
- Fine-tuning catches a non-finite loss at the step it appears and retries that model once on CPU (a safety net for any future divergence), recording the device used in the manifest. The risk regression drops any surrogate whose similarity features are non-finite before fitting, since scikit-learn rejects NaN inputs.
- Node modules import their annotation types (`pandas`, `matplotlib.figure.Figure`, `typing.Any`) at module scope instead of under `if TYPE_CHECKING:`. Kedro 1.4's parameter-type validation calls `typing.get_type_hints` on every node function at startup, which evaluates the annotations against module globals; with the imports deferred the names did not resolve and Kedro logged a `kedro.validation.type_extractor` warning per node. Ruff's `TC001/2/3` are ignored for `pipelines/**/nodes.py` to keep these imports at runtime.

### Notes

- All seven domain pipelines and the `transfer_risk.lib` core are implemented; the coverage gate is 90% on `lib`. The project still **measures and compares** transferability risk; it never **certifies** robustness (Vassilev 2025).
- The heavy ML stack (torch, transformers 5, datasets, scikit-learn, TextAttack via the turn1a fork, sentence-transformers) runs in one main environment on Python 3.13 with MPS — no per-stage subenvs.
